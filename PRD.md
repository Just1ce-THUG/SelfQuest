# PRD: MVP мобильного приложения SelfQuest

## 1. Название проекта

SelfQuest

Название рабочее и может быть изменено позже.

## 2. Краткое описание

SelfQuest — мобильное приложение для создания и прохождения личных челленджей.

Пользователь может создать челлендж, задать срок, фиксировать прогресс и видеть, насколько он продвинулся к цели.

MVP поддерживает три типа челленджей:

1. Числовой челлендж.
2. Ежедневный челлендж.
3. Проектный челлендж.

Приложение работает локально на устройстве. Авторизация, облако, ИИ, социальные функции и уведомления не входят в MVP.

## 3. Цель MVP

Цель MVP — проверить базовую механику:

пользователь создаёт челлендж → фиксирует прогресс → видит процент выполнения → продолжает пользоваться приложением.

MVP должен быть простым, стабильным и понятным. Первая версия не должна превращаться в перегруженный комбайн.

## 4. Целевая аудитория

Приложение предназначено для пользователей, которые любят личные челленджи и хотят отслеживать прогресс по целям.

Основные сценарии:

1. Сделать определённое количество действий за период.
2. Выполнять действие каждый день.
3. Разбить большой проект на этапы и шаги.
4. Видеть прогресс в процентах.
5. Возвращаться к челленджу каждый день.

## 5. Основные типы челленджей

### 5.1 Числовой челлендж

Числовой челлендж — это челлендж, где есть общая числовая цель.

Примеры:

* сделать 1000 отжиманий за 30 дней;
* пройти 100 километров за месяц;
* прочитать 500 страниц;
* потратить 20 часов на изучение темы.

Пользователь задаёт:

* название;
* описание;
* общее целевое число;
* единицу измерения;
* количество дней;
* дату начала.

Пользователь ежедневно вводит число, которое он сделал за день.

Приложение показывает:

* сколько сделано всего;
* сколько осталось;
* процент выполнения;
* историю прогресса по дням.

### 5.2 Ежедневный челлендж

Ежедневный челлендж — это челлендж, где нужно каждый день выполнять одно действие.

Примеры:

* читать по 5 страниц каждый день;
* делать зарядку каждый день;
* не есть сладкое 14 дней;
* заниматься английским каждый день.

Пользователь задаёт:

* название;
* описание;
* ежедневное действие;
* количество дней;
* дату начала.

Пользователь отмечает дни как выполненные или невыполненные.

Приложение показывает:

* календарь челленджа;
* выполненные дни;
* невыполненные дни;
* процент выполнения;
* текущий статус.

### 5.3 Проектный челлендж

Проектный челлендж — это челлендж для большой цели, которую нужно разбить на этапы и шаги.

Примеры:

* создать мобильное приложение;
* подготовиться к экзамену;
* собрать портфолио;
* дойти до жима 100 кг;
* закончить учебный проект.

Пользователь задаёт:

* название;
* описание;
* количество дней;
* дату начала.

После создания пользователь сам добавляет этапы и шаги.

Пример:

Челлендж: “Создать приложение”.

Этап 1: “Продумать продукт”.
Шаги:

* описать идею;
* определить MVP;
* написать PRD.

Этап 2: “Сделать интерфейс”.
Шаги:

* сделать главный экран;
* сделать экран создания челленджа;
* сделать экран деталей челленджа.

Этап 3: “Реализовать данные”.
Шаги:

* создать базу данных;
* сохранить челленджи;
* сохранить прогресс.

Пользователь отмечает шаги выполненными. Прогресс считается по выполненным нижним шагам.

## 6. Функции MVP

В MVP входят:

1. Главный экран.
2. Список челленджей.
3. Создание челленджа.
4. Детальный экран челленджа.
5. Редактирование челленджа.
6. Удаление челленджа.
7. Числовой челлендж.
8. Ежедневный челлендж.
9. Проектный челлендж.
10. Прогресс-бар.
11. Процент выполнения.
12. История прогресса.
13. Простое календарное отображение для ежедневного челленджа.
14. Этапы и шаги для проектного челленджа.
15. Локальное хранение данных.
16. Сохранение данных после перезапуска приложения.

## 7. Что не входит в MVP

В MVP не входят:

1. Искусственный интеллект.
2. Авторизация.
3. Регистрация.
4. Облачная синхронизация.
5. Уведомления.
6. Социальные челленджи.
7. Друзья.
8. Команды.
9. Рейтинги.
10. Подписки и платежи.
11. Готовые шаблоны челленджей.
12. Фото- или видео-доказательства.
13. Виджеты.
14. Сложная статистика.
15. Экспорт данных.
16. Глубокая вложенность проектных задач больше двух уровней.

## 8. Главный экран

Главный экран показывает общее состояние приложения.

Если челленджей нет, пользователь видит пустое состояние:

“Пока нет челленджей. Создай первый челлендж и начни движение к цели.”

На экране есть кнопка:

“Создать челлендж”.

Если челленджи есть, главный экран показывает:

1. Количество активных челленджей.
2. Список активных челленджей.
3. Карточки челленджей.
4. Кнопку создания нового челленджа.

Карточка челленджа содержит:

* название;
* тип;
* процент выполнения;
* прогресс-бар;
* дату окончания;
* статус.

## 9. Экран списка челленджей

Экран списка показывает все челленджи.

Фильтры:

1. Все.
2. Активные.
3. Завершённые.

Каждый челлендж отображается карточкой.

Карточка содержит:

* название;
* тип;
* прогресс;
* статус;
* срок;
* краткую информацию по типу челленджа.

## 10. Экран создания челленджа

Пользователь нажимает “Создать челлендж”.

Сначала выбирает тип:

1. Числовой.
2. Ежедневный.
3. Проектный.

После выбора типа отображается соответствующая форма.

Общие поля:

* название;
* описание;
* дата начала;
* количество дней.

Поля числового челленджа:

* целевое число;
* единица измерения.

Поля ежедневного челленджа:

* ежедневное действие.

Поля проектного челленджа:

* название;
* описание;
* дата начала;
* количество дней.

Этапы и шаги для проектного челленджа можно добавить после создания.

## 11. Экран числового челленджа

Экран показывает:

1. Название.
2. Описание.
3. Прогресс-бар.
4. Процент выполнения.
5. Сделано всего.
6. Осталось.
7. Дней осталось.
8. Поле ввода результата за сегодня.
9. Кнопку “Добавить прогресс”.
10. Историю записей.
11. Кнопку редактирования.
12. Кнопку удаления.

Пример:

“Сделано: 250 / 1000 отжиманий”

“Осталось: 750 отжиманий”

“Прогресс: 25%”

## 12. Экран ежедневного челленджа

Экран показывает:

1. Название.
2. Описание.
3. Ежедневное действие.
4. Прогресс-бар.
5. Процент выполнения.
6. Сетку дней.
7. Кнопку “Выполнено сегодня”.
8. Возможность вручную изменить статус дня.
9. Кнопку редактирования.
10. Кнопку удаления.

Статусы дня:

* выполнено;
* не выполнено;
* без отметки.

Для MVP календарь может быть простой сеткой дней челленджа, а не полноценным календарём по месяцам.

## 13. Экран проектного челленджа

Экран показывает:

1. Название.
2. Описание.
3. Прогресс-бар.
4. Процент выполнения.
5. Список этапов.
6. Шаги внутри этапов.
7. Кнопку “Добавить этап”.
8. Кнопку “Добавить шаг”.
9. Чекбоксы выполнения шагов.
10. Возможность редактировать этапы и шаги.
11. Возможность удалять этапы и шаги.
12. Кнопку редактирования челленджа.
13. Кнопку удаления челленджа.

В MVP проектный челлендж имеет два уровня:

* этап;
* шаг.

Более глубокая вложенность не нужна для первой версии.

## 14. Редактирование

Пользователь может редактировать:

* название;
* описание;
* дату начала;
* количество дней;
* целевое число для числового челленджа;
* единицу измерения;
* ежедневное действие;
* этапы проектного челленджа;
* шаги проектного челленджа.

После редактирования прогресс должен пересчитываться, если изменились параметры цели.

## 15. Удаление

Пользователь может удалить челлендж.

Перед удалением нужно показать подтверждение:

“Удалить челлендж? Все данные прогресса будут удалены.”

Кнопки:

* “Отмена”;
* “Удалить”.

Если пользователь удаляет этап проектного челленджа, внутри которого есть шаги, нужно показать подтверждение:

“Удалить этап и все вложенные шаги?”

## 16. Завершение челленджа

Челлендж получает статус `completed`, если:

1. Числовой челлендж достиг 100%.
2. Ежедневный челлендж имеет все дни со статусом “выполнено”.
3. Проектный челлендж имеет все нижние шаги выполненными.

Завершённый челлендж должен отображаться в фильтре “Завершённые”.

## 17. Расчёт прогресса

### 17.1 Числовой челлендж

Формула:

`progressPercent = totalCompleted / targetValue * 100`

Где:

`totalCompleted` — сумма всех записей прогресса.

Если прогресс больше 100%, показывать 100%.

Если остаток меньше 0, показывать 0.

### 17.2 Ежедневный челлендж

Формула:

`progressPercent = completedDays / durationDays * 100`

Где:

`completedDays` — количество дней со статусом “выполнено”.

Если прогресс больше 100%, показывать 100%.

### 17.3 Проектный челлендж

Формула:

`progressPercent = completedLeafItems / totalLeafItems * 100`

Leaf item — нижний элемент структуры.

Если у этапа есть шаги, считаются шаги.

Если у этапа нет шагов, сам этап считается leaf item.

Все leaf items имеют одинаковый вес.

Если totalLeafItems равно 0, прогресс равен 0%.

## 19. Технический стек MVP

### 19.1 Основной стек

* React Native;
* Expo;
* TypeScript;
* Expo Router;
* Expo SQLite;
* Drizzle ORM;
* Zustand;
* React Hook Form;
* Zod;
* NativeWind;
* Lucide React Native;
* Jest;
* React Native Testing Library.

### 19.2 Почему выбран этот стек

React Native и Expo выбраны, потому что позволяют разрабатывать мобильное приложение для Android и iOS из одной кодовой базы.

TypeScript выбран, потому что в приложении есть разные типы челленджей и разные структуры данных. Типизация должна снижать количество ошибок при разработке.

Expo Router выбран для навигации между экранами.

Expo SQLite выбран для локального хранения челленджей и прогресса на устройстве.

Drizzle ORM выбран для более удобной и типизированной работы с SQLite.

Zustand выбран для лёгкого управления состоянием интерфейса.

React Hook Form и Zod выбраны для форм и валидации данных.

NativeWind выбран для быстрой стилизации интерфейса.

Lucide React Native выбран для иконок.

Jest и React Native Testing Library выбраны для тестирования бизнес-логики и базового поведения компонентов.

### 19.3 Что не используется в MVP

В MVP не используются:

* backend;
* Supabase;
* Firebase;
* OpenAI API;
* авторизация;
* cloud sync;
* push notifications;
* payments.

## 21. Валидация

Общие правила:

1. Название челленджа обязательно.
2. Количество дней должно быть больше 0.
3. Дата начала обязательна.
4. Целевое число в числовом челлендже должно быть больше 0.
5. Единица измерения обязательна для числового челленджа.
6. Ежедневное действие обязательно для ежедневного челленджа.
7. Проектный челлендж может быть создан без этапов, но его прогресс будет 0%.

### 22.1 Нет челленджей

Кнопка:

“Создать челлендж”.

### 22.2 Нет прогресса

“Добавь первый этап, чтобы разбить проект на понятные части.”

“Добавить этап”.

## 23. Критерии готовности MVP

MVP считается готовым, если пользователь может:

1. Открыть приложение.
2. Увидеть главный экран.
3. Создать числовой челлендж.
4. Добавить прогресс в числовой челлендж.
5. Увидеть процент и остаток числового челленджа.
6. Создать ежедневный челлендж.
7. Отметить день выполненным.
8. Увидеть календарь ежедневного челленджа.
9. Создать проектный челлендж.
10. Добавить этапы.
11. Добавить шаги внутри этапов.
12. Отметить шаги выполненными.
13. Увидеть процент проектного челленджа.
14. Редактировать челлендж.
15. Удалить челлендж.
16. Закрыть приложение.
17. Открыть приложение снова и увидеть сохранённые данные.

## 24. Минимальные тесты

Для MVP нужно покрыть тестами хотя бы расчёт прогресса.

Тесты должны проверять:

1. Числовой прогресс считается правильно.
2. Числовой прогресс не превышает 100%.
3. Остаток не становится отрицательным.
4. Ежедневный прогресс считается по выполненным дням.
5. Проектный прогресс считается по нижним элементам.
6. Проектный прогресс равен 0%, если нет этапов и шагов.

## 26. Главный принцип разработки

Сначала нужно реализовать простое локальное приложение.

Порядок разработки:

1. Создать структуру проекта.
2. Настроить навигацию.
3. Создать базовые UI-компоненты.
4. Создать модели данных.
5. Реализовать локальное хранение.
6. Реализовать создание челленджа.
7. Реализовать список челленджей.
8. Реализовать детали числового челленджа.
9. Реализовать детали ежедневного челленджа.
10. Реализовать детали проектного челленджа.
11. Реализовать редактирование и удаление.

Главная задача MVP — не максимальное количество функций, а рабочая и понятная базовая механика.
